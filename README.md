# Case Definition and Configuration Guide

This document describes how to configure and run simulations using the DGTD Model Order Reduction (MOR) solver. It outlines data prerequisites, expected file structures, and the detailed anatomy of the JSON configuration file.

## Theoretical Framework & Mathematical Formulation

The MOR solver is built to project and accelerate linear time-invariant (LTI) state-space systems governing electromagnetic or physical simulations. The framework operates on the following continuous formulation:

$$\dot{x}(t) = Ax(t) + B(Tu(t))$$

Where:
* $x(t)$ represents the **system state vectors** (e.g., full-order discrete electromagnetic field coefficients).
* $u(t)$ represents the **external boundary or forcing inputs** injected into the domain.
* $A$ is the global full-order system evolution operator.
* $B$ and $T$ are the specialized structural operators responsible for mapping and handling Total-Field Scattered-Field (TFSF) or incident wave source conditions.

The temporal evolution of the reduced state-space variables is advanced stably and efficiently using a low-storage **4th-order Runge-Kutta (LSERK4)** integration scheme.

## 1. Data Prerequisites & Directory Structure

The MOR solver is designed to ingest high-fidelity data generated by the **OpenSEMBA DGTD** full-order solver. 

* **Default Source Repository:** [OpenSEMBA/dgtd](https://github.com/OpenSEMBA/dgtd).

### Full-Order Simulation Strategies
To deploy the MOR pipeline effectively, full-order simulations are leveraged differently depending on their role in the workflow:

1. **Training Case (Snapshot Generation):** The full-order simulation must be executed over a sufficiently long time window to collect rich field distributions. The goal is to gather comprehensive information regarding the system's underlying dynamics and **eigenvalues** to build a faithful, highly accurate reduced projection basis ($U_r$) via Proper Orthogonal Decomposition (POD).
2. **Target Evolution Case:** Unlike the training stage, the target model to evolve **does not require a lengthy full-order execution**. Structural operators ($A$, $B$, $T$) are time-invariant and completely assembled right during the setup phase / first iteration of the solver. 
   The only critical runtime information required from this target case is the discrete boundary input sequence ($u$) at each time step. *Ideally, these forcing inputs should be obtainable via analytical models, mathematical expressions, or external streams, bypassing the need to execute the full-order simulation entirely.*

### Custom Data Sources
If you wish to utilize data from an alternative electromagnetic solver or experimental framework, the data **must match the exact naming conventions and folder layout** produced by OpenSEMBA DGTD. 

1. **Operators:** Structural operators ($A$, $B$, $T$) must be formatted and saved as **MFEM Compressed Sparse Row (CSR)** matrix files.
2. **Snapshots and Inputs:** Fields or probe files must adhere to the standard snapshot nomenclature. The formatting rule for these files requires:
   * **First row:** The absolute timestamp (`time`).
   * **Second row:** The total size/order of the system vector (`size`).
   * **Subsequent lines:** The numerical spatial coefficients.

By satisfying this format layout, you can seamlessly direct the MOR solver to your alternative dataset simply by modifying the global `exports_dir` variable in your configuration JSON.

### Expected Directory Layout
The solver deduces target paths automatically assuming the following subdirectory convention inside your specified `exports_dir`:

```text
<exports_dir>/
├── Operators/
│   └── <evolve_case_name>/
│       ├── <evolve_case_name>_global.csr          # System Evolution Matrix (A)
│       ├── <evolve_case_name>_tfsf.csr            # TFSF Operator Matrix (B)
│       └── <evolve_case_name>_tfsf_mapping.csr    # TFSF Mapping Matrix (T)
└── single-core/
    ├── <train_case_name>/
    │   └── MORStateProbes/
    │       └── MORState/                         # Training snapshot files
    └── <evolve_case_name>/
        └── MORStateProbes/
            └── MORState/                         # Forcing boundary inputs (u_0, etc.)
```

## 2. JSON Configuration File Reference

The execution pipeline is entirely controlled via a JSON layout. Below is an example configuration showcasing a scenario where the Reduced Basis is trained on one case and evolved over a different target case:

```json
{
  "exports_dir": "external/dgtd/Exports",
  
  "training": {
    "case_name": "2D_TFSF_MOR_training",
    "energy_threshold": 0.999999999999,
    "model_order": -1,
    "n_blocks": 1,
    "k_svd_max": 1000,
    "filter_low_energy": false,
    "filter_min_energy": 1e-6
  },
  
  "model_to_evolve": {
    "case_name": "2D_TFSF_MOR_evolve"
  },
  
  "solver_options": {
    "dt": 1e-2,
    "final_time": 10.0,
    "case_name": "2D_TFSF_MOR_ROM_Output"
  }
}
```

### Parameter Reference Table

#### Global Scope
| Variable | Type | Status | Description |
| :--- | :--- | :--- | :--- |
| `exports_dir` | String | **Mandatory** | Relative or absolute path pointing to the root directory where the DGTD dataset structure is located. |

#### `training` Block
Contains settings related to Proper Orthogonal Decomposition (POD) and randomized Singular Value Decomposition (SVD) on simulation snapshots.

| Variable | Type | Status | Default | Description |
| :--- | :--- | :--- | :--- | :--- |
| `case_name` | String | **Mandatory** | *N/A* | Name of the subfolder under `<exports_dir>/single-core/` from which the training snapshots matrix will be loaded. |
| `energy_threshold` | Double | Optional | `1.0 - 1e-12` | Cumulative singular value energy content to retain during SVD subspace truncation. |
| `model_order` | Integer | Optional | `-1` | Explicit size/dimension of the reduced basis subspace. If set to `-1`, it triggers **Automatic Rank Truncation** based on the `energy_threshold`. |
| `n_blocks` | Integer | Optional | `1` | Number of blocks utilized during incremental POD steps. |
| `k_svd_max` | Integer | Optional | `1000` | Upper bound for the truncation rank guard. If `model_order > 0`, defaults safely to `model_order + 20`. |
| `filter_low_energy`| Boolean | Optional | `false` | Enables manual stripping of extremely low energy singular components. |
| `filter_min_energy`| Double | Optional | `1e-6` | Energy floor value used only if `filter_low_energy` is set to `true`. |

#### `model_to_evolve` Block
Defines the target scenario to project and simulate inside the reduced subspace space.

| Variable | Type | Status | Description |
| :--- | :--- | :--- | :--- |
| `case_name` | String | **Mandatory** | Folder identifier under `Operators/` and `single-core/` containing the full-order matrix systems ($A, B, T$) and forcing boundary terms that will be processed in the timeline. |

#### `solver_options` Block
Configures time integration via the Low-Storage Runge-Kutta 4th order (LSERK4) routine.

| Variable | Type | Status | Default | Description |
| :--- | :--- | :--- | :--- | :--- |
| `dt` | Double | **Mandatory** | *N/A* | Time-step size for the solver. <br>`dt` **must exactly match** the time-step size used during the original full-order DGTD dataset generation to preserve phase and system stability. |
| `final_time` | Double | **Mandatory** | *N/A* | Absolute simulation end-time marker. |
| `case_name` | String | Optional | `""` | The folder identifier under `Exports/<case_name>/` where the time evolution data and reconstructed full-order fields will be written. If omitted or left empty, all file logging and exporting are silently bypassed to optimize execution speed. |
